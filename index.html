Emergent Workflow

When you go back to it
Emergent → new chat→ Pull from Preview-Test repo→ Build feature→ Emergent pushes to Preview-Test→ Check Vercel preview→ GitHub → Squash merge to main
Why squash merge from Emergent? Emergent makes lots of small auto-commits with generated names. Squash merge bundles all of them into one clean commit on main so your history stays readable.
After Emergent pushes, sync Cursor before editing again: git checkout Preview-Test then git pull origin Preview-Test — so Cursor always has the latest before you start.

Tab at the top of the repo. Click it, then click the green "New pull request" button.

Set the correct branches
base: main ← compare: Preview-Test
This means "merge Preview-Test INTO main".

Review the diff, write a title

Click the dropdown arrow next to the merge button → choose Squash and Merge. One feature = one clean commit on main.

Common Git Errors & Fixes

When it all goes wrong
ERROR rejected — push declined, remote contains work that you do not have locally

GitHub has commits your local doesn't. Usually happens when Emergent or a GitHub action pushed something after you last pulled.

fix — pull first, then push
git pull origin Preview-Test --rebasecopy
git push origin Preview-Testcopy
ERROR CONFLICT (content): Merge conflict in src/components/something.jsx

Two sources edited the same file. Git doesn't know which to keep. Cursor will highlight the conflict markers inside the file.

fix — open the file, pick your version, then commit

Look for <<<<<<< HEAD markers in the file. Delete the version you don't want and the markers themselves. Then:

git add .copy
git commit -m "fix: resolve merge conflict"copy
git push origin Preview-Testcopy
ERROR HEAD detached at a3f92b1

You're looking at an old commit but not on any branch. Any new commits here won't be saved to Preview-Test or main.

fix — get back on your branch
git checkout Preview-Testcopy
ERROR Your branch and 'origin/Preview-Test' have diverged

Local and remote went in different directions. This is exactly what caused your situation today.

fix A — trust your local, overwrite remote
git push origin Preview-Test --forcecopy
fix B — trust remote, overwrite local
git reset --hard origin/Preview-Testcopy
ERROR nothing to commit, working tree clean — but site not updated on GitHub

Git thinks nothing changed. The files you edited may have been untracked or you may be in the wrong folder.

fix — verify what git actually sees
git statuscopy
git log --oneline origin/Preview-Test -3copy

If the top commit matches what you want, GitHub already has your code. Issue is your hosting platform — check Vercel.

POWERSHELL A parameter cannot be found that matches parameter name 'rf'

You used a bash command in PowerShell. rm -rf doesn't exist here.

fix — use PowerShell syntax
Remove-Item -Recurse -Force folder-name/copy
06

Vercel Troubleshooting

When the live site is wrong
Build failed

Vercel → your project → failed deployment → click "View build logs". Look for the red error at the bottom. Usually a missing import or broken syntax. Fix in Cursor, push to Preview-Test, Vercel auto-retries.

Pushed to GitHub but Vercel didn't deploy

Vercel may have lost the GitHub connection. Go to Vercel → Settings → Git → verify the correct repo and branch is connected. Also confirm a deployment was triggered after your push.

Wrong commit is active / old code showing

In Vercel deployments list, find the correct deployment by commit message → click it → click "Promote to Production". Instantly makes that exact commit live — no Git commands needed.

Environment variable missing

Vercel doesn't read your local .env file. Go to Vercel → Settings → Environment Variables → add each variable manually. Redeploy after adding them.

Site shows old version

Open your live URL in an incognito window. If it shows new version there, it's just browser cache. Hard refresh: Ctrl+Shift+R (Windows) or Cmd+Shift+R (Mac).

Preview vs Production

Preview-Test branch → Vercel Preview URL (for testing). Main branch → Vercel Production URL (your live site). Always verify on Preview URL before merging to main.

Golden Rules

Never commit directly to mainAlways go through Preview-Test. Main = production = live site.
Tag every big wingit tag v1.x-name after every feature that works perfectly.
Pull before you buildAlways git pull before starting new work. Avoids diverged branch mess.
Check Vercel before mergingNever merge blind into production. Verify the preview URL first.
Force push = last resortOnly use --force in emergencies. It rewrites history permanently.
Squash merge from EmergentKeeps main history clean. One feature = one commit on main.
reflog is your time machineYou can ALWAYS recover. Every action is logged. Never panic.
Test in incognito firstBefore assuming a deploy failed, check private window — it's often just browser cache.

Git Terms — Plain English

commitA saved snapshot of your code
pushSend local code up to GitHub
pullGet latest code from GitHub to your computer
branchA parallel version of your code
mergeCombine two branches together
squashBundle many commits into one clean commit
tagA permanent named bookmark on a commit
force pushOverride GitHub with your local — use carefully
stashTemporarily hide uncommitted changes
reflogYour Git time machine — every action ever taken
HEADWhere you are right now in Git history
originThe remote copy of your repo on GitHub
detached HEADBrowsing old commit, not on any branch — run git checkout Preview-Test to escape
rebaseRewrite commit history to be cleaner — use only when Claude tells you to
